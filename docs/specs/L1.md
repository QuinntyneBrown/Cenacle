# L1 — High-Level Requirements

**Product:** Cenacle — a browser-native prayer gathering space.
**Source:** `docs/cenacle-prd.md` · **UI reference:** `docs/mocks/` (31 screens) · `docs/mocks/STYLE-GUIDE.md`

These high-level requirements state *what* the system must do. Each L2 in `L2.md` traces
back to exactly one L1 here. Numbering is stable — never renumber; add with the next
free number.

Scope discipline (from the PRD): a capability belongs in v1 only if it strengthens
**presence**, **privacy**, or the **sanctuary feel**. v1 non-goals — user accounts / social
graph, recorded VOD, text chat, payments, native mobile, moderation tooling, and broadcast
scale — are out of scope and must not be built.

---

## Subsystem A — Presence (the live gathering)

## L1-001: Host a live gathering
The system must let a person turn a single browser tab into a broadcast origin: name a
gathering, choose camera and microphone, preview themselves privately, choose on-device
options, and go live — with no plugin, no encoder software, and no RTMP.

## L1-002: Join a live gathering
The system must let a person join an existing gathering by room code or invite link, preview
and configure their own camera and microphone in a green room, grant device permissions, and
enter the live room.

## L1-003: Sub-second live presence
The system must capture, encode, transport, decode, and render participant audio and video
with sub-second, frame-accurate presence so a gathering feels co-located rather than buffered.

## L1-004: In-room presence and controls
The system must present the live room — active speaker, participant filmstrip, and a roster of
who is present — and give each person local controls over their microphone, camera, captions,
and access to the companion, distinguished by role (host vs participant).

## L1-005: Corporate reactions ("Amen")
The system must let anyone in the room send a lightweight reaction (amen / raised hand) that is
surfaced to everyone in the room near-instantly, and must show the room responding as one.

## L1-006: Invitations and room access
The system must give a host a way to invite others — a shareable link, a short room code, and a
QR code — and must let those artifacts be copied for sharing.

## L1-007: Room lifecycle and resilience
The system must manage a room's lifecycle: a participant leaving (and rejoining while open), a
host ending the gathering for everyone, recovery from a dropped connection, and clear handling
when a room is full, has ended, or cannot be found.

## Subsystem B — Word (the private companion)

## L1-008: Scripture surfacing
The system must let a person name a theme in plain words and surface a fitting passage retrieved
from a local verse index on the device — passages retrieved, never generated — including a clear
"no passage matched" outcome.

## L1-009: Private lament journal
The system must provide a journal where a person writes freely and can request a gentle,
grounded reflection, where entries and reflections are created and stored only on the device and
are never transmitted.

## L1-010: Live captions
The system must transcribe spoken prayer and teaching on the device in real time for
accessibility, displaying captions in the room, with a selectable caption language and a toggle.

## L1-011: On-device intelligence foundation
The system must run all Word-subsystem intelligence (reflections, Scripture matching, captions)
locally via the browser's on-device AI, including feature-detecting availability, guiding a
one-time model download, and letting the model be managed.

## Subsystem C — Sanctuary (the ambient space)

## L1-012: Ambient sanctuary visuals
The system must render a GPU-native ambient atmosphere for the room — light, depth, and motion —
in place of a flat frame, at a smooth frame rate, as an optional layer.

## L1-013: Audio-reactive worship visuals
The system must let the ambient visuals respond in real time to the live worship audio, so the
space visibly moves with the music.

## L1-014: Floating prayer companion
The system must let a person pop the gathering into an always-on-top floating window that stays
present over other applications and tabs while they read or work elsewhere, and return it to the
full room without dropping anyone.

## Cross-cutting concerns

## L1-015: Settings and device management
The system must provide a settings surface to configure devices (camera, microphone, speaker),
presence (codec, latency target), captions and Word, and sanctuary visuals, with all preferences
stored on the device only.

## L1-016: Capability detection and graceful degradation
The system must feature-detect every advanced browser capability it depends on and degrade
gracefully where one is missing — hiding or substituting the affected feature with a clear
explanation — never blocking the whole experience when only part is unsupported.

## L1-017: Privacy by architecture
The system must guarantee, by architecture rather than policy, that private content (journal
entries, reflections, Scripture themes, captions) never leaves the device. Where an on-device
capability is unavailable, the affected feature must be hidden — never rerouted to a server. The
guarantee must be independently verifiable (zero outbound requests).

## L1-018: Security and data protection
The system must protect the gathering and its data: authenticated and encrypted transport,
authorized room access resistant to enumeration and abuse, validation and safe rendering of all
user-supplied input, and protection of locally stored data — consistent with the OWASP Top 10.

## L1-019: Performance and scalability
The system must meet its presence latency target, render visuals smoothly, load quickly, use
device resources efficiently, and operate within a small-room topology (fan-out to large scale is
explicitly out of scope for v1).

## L1-020: Responsive and accessible experience
The system must work across all viewport sizes (extra-small through extra-large) and meet
accessibility expectations — keyboard operability, screen-reader semantics, live-region
announcements for captions and presence, reduced-motion support, and sufficient contrast — across
both the light ("manuscript") and night ("upper room") visual registers.

## L1-021: Reliability, observability, and operations
The system must fail safely and observably: no silent failures in the client, privacy-preserving
operational telemetry that never captures prayer content or personal identifiers, and a
monitored HTTP/3 origin for the live transport.
