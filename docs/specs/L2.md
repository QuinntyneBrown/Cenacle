# L2 — Detailed Requirements

Each requirement traces to exactly one L1 in `L1.md`. Acceptance criteria use Given/When/Then so
they translate directly into acceptance tests. Numbering is stable — never renumber.

**Responsive breakpoints** (referenced throughout): XS `<576px` · SM `≥576px` · MD `≥768px` ·
LG `≥992px` · XL `≥1200px`.

**ATDD convention (reminder):** every acceptance test file must carry a trace header naming the
L2(s) it covers, e.g.
```
// Acceptance Test
// Traces to: L2-005, L2-011
// Description: Host go-live publishes an encoded WebTransport stream
```
The trace comment is not optional — it is how coverage is verified.

## Subsystem A — Presence

## L2-001: Name the gathering
**Traces to:** L1-001

The host setup must let the host name the gathering. The field defaults to a sensible value
(e.g. "Vespers"), accepts 1–60 characters, and must be present before going live.

**Acceptance Criteria:**
1. Given the host setup screen, when it loads, then the name field is pre-filled with a non-empty default and focus can reach it via keyboard.
2. Given the name field is empty or contains only whitespace, when the host attempts to go live, then go-live is blocked and an inline hint states a name is required.
3. Given a name longer than 60 characters, when the host types, then input is capped at 60 characters.
4. Given a name containing markup or control characters, when it is later rendered in the room and invite surfaces, then it is displayed as inert text (see L2-073).

---

## L2-002: Select camera and microphone at setup
**Traces to:** L1-001

The host setup must enumerate available cameras and microphones and let the host choose which to
use before going live.

**Acceptance Criteria:**
1. Given granted device permissions, when the setup screen loads, then the camera and microphone selects list the available devices and show the current selection.
2. Given the host picks a different camera or microphone, when the selection changes, then the self-preview updates to reflect the newly selected device.
3. Given no camera or no microphone is available, when the setup screen loads, then the affected control shows an empty/disabled state with guidance rather than an error crash.

---

## L2-003: Private self-preview before going live
**Traces to:** L1-001

The host setup must show a private self-preview (night register) labelled as visible only to the
host, including a live microphone-level indicator, and must state that nothing is recorded.

**Acceptance Criteria:**
1. Given a selected camera, when the setup screen is shown, then a self-preview renders the host's own video and is labelled "Preview · only you".
2. Given a selected microphone receiving sound, when the host speaks, then a microphone-level indicator moves in response.
3. Given the self-preview, when it is displayed, then copy explicitly states nothing is recorded and no media has been transmitted.

---

## L2-004: On-device options at setup
**Traces to:** L1-001

The host setup must let the host toggle live captions (on-device) and living sanctuary visuals
before going live, defaulting both on, each with a plain description of what it does.

**Acceptance Criteria:**
1. Given the setup screen, when it loads, then the "Live captions" and "Living sanctuary visuals" toggles are present and default to on.
2. Given the host disables a toggle and goes live, when the room opens, then that feature is inactive in the room.
3. Given the on-device model is unavailable (L2-049), when the setup screen loads, then the captions toggle is hidden rather than shown as an option that would reroute to a server.

---

## L2-005: Go live as a broadcast origin
**Traces to:** L1-001

Going live must acquire the selected devices, encode media with WebCodecs, and publish it to the
room over WebTransport, transitioning the host into the live room as host.

**Acceptance Criteria:**
1. Given a valid name and selected devices, when the host activates "Go live", then the host enters the live room shown as host and broadcasting.
2. Given the host has gone live, when publishing starts, then media is encoded via WebCodecs and published over WebTransport (no plugin, no external encoder, no RTMP).
3. Given device acquisition fails at go-live, when the error occurs, then the host is shown the matching recovery state (L2-068) instead of a blank room.

---

## L2-006: Join by room code
**Traces to:** L1-002

The system must let a person join by entering a six-character room code, case-insensitive on
entry and normalized to uppercase.

**Acceptance Criteria:**
1. Given the join dialog, when a person types a code, then input is limited to six characters and auto-capitalized.
2. Given a valid six-character code for an open room, when the person submits, then they proceed to the green room for that gathering.
3. Given a code that is not six characters, when the person submits, then submission is blocked with a hint that codes are six characters.
4. Given a well-formed code that matches no open room, when the person submits, then the "room not found" state (L2-032) is shown.

---

## L2-007: Join by invite link
**Traces to:** L1-002

The system must let a person join by opening an invite link or pasting one into the join dialog,
resolving it to the same green-room entry flow as a code.

**Acceptance Criteria:**
1. Given an invite link of the form `.../r/{CODE}`, when it is opened, then the person is taken toward the green room for that room (subject to permissions).
2. Given the join dialog, when the person pastes an invite link, then the room code is extracted and the join proceeds as if the code were entered.
3. Given a malformed or foreign URL pasted as an invite, when the person submits, then it is rejected with a hint rather than navigating away.

---

## L2-008: Green room (self-preview and pre-join configuration)
**Traces to:** L1-002

Before entering, the joiner must see a green-room self-preview and be able to set their display
name and pre-toggle microphone and camera; those pre-join states must carry into the room.

**Acceptance Criteria:**
1. Given the green room, when it loads, then it shows a self-preview, a name field, and microphone/camera pre-toggles reflecting their current state.
2. Given the joiner sets microphone off and camera on in the green room, when they enter, then they enter with microphone muted and camera live.
3. Given the green room, when displayed, then it states that camera and mic go live the moment the person enters.
4. Given an empty name, when the joiner enters, then a non-empty display name is used (name is optional; a fallback label is applied).

---

## L2-009: Camera and microphone permission primer
**Traces to:** L1-002

The system must show a plain-language primer explaining why camera and microphone are needed and
that nothing is recorded, before triggering the browser's native permission prompt.

**Acceptance Criteria:**
1. Given a person about to enter who has not granted permissions, when the primer is shown, then it explains the camera and microphone uses and states nothing is recorded.
2. Given the primer, when the person chooses "Allow camera & mic", then the browser's native permission prompt is requested.
3. Given the person chooses "Not now", when they decline, then they are returned to a safe prior screen without media being acquired.
4. Given permission is denied at the browser prompt, when the denial is detected, then the permission-denied recovery state (L2-068) is shown.

---

## L2-010: Enter the live room
**Traces to:** L1-002

Entering must subscribe to the room's media stream, decode it, and render the gathering with the
joiner placed among the participants.

**Acceptance Criteria:**
1. Given a joiner who has entered, when the room loads, then the host appears on the main stage and other participants appear in the filmstrip, including the joiner as "You".
2. Given the room is at capacity, when the joiner attempts to enter, then the "room full" state (L2-031) is shown instead.
3. Given entry succeeds, when the stream is established, then decoded remote video renders within the presence latency target (L2-013).

---

## L2-011: Publish encoded media over WebTransport
**Traces to:** L1-003

The system must publish captured media as WebCodecs-encoded frames over a WebTransport (HTTP/3)
connection to the room.

**Acceptance Criteria:**
1. Given a live publisher, when frames are captured, then they are encoded with WebCodecs and sent over a WebTransport connection.
2. Given the negotiated codec, when encoding, then the configured codec (H.264 or VP9 per L2-014) is used.
3. Given the WebTransport connection drops, when publishing fails, then the client transitions to the reconnect flow (L2-030) and stops sending media.

---

## L2-012: Subscribe, decode, and render with frame-accurate sync
**Traces to:** L1-003

The system must pull each remote participant's stream, decode it with WebCodecs, and render to a
canvas/WebGPU surface synchronized to frame callbacks.

**Acceptance Criteria:**
1. Given a subscribed stream, when frames arrive, then they are decoded and rendered on a per-frame callback so motion is smooth and tearing-free.
2. Given multiple remote participants, when they are visible, then each renders in its own tile without blocking the others.
3. Given a participant stops sending (mutes camera), when their video ends, then their tile shows a camera-off indicator rather than a frozen last frame indefinitely.

---

## L2-013: Presence latency target and readout
**Traces to:** L1-003

Glass-to-glass latency must be under 400 ms on a LAN-class connection, and the room must display
a live latency readout.

**Acceptance Criteria:**
1. Given a LAN-class connection, when two participants are live, then measured glass-to-glass latency is under 400 ms.
2. Given a live room, when it is displayed, then a latency readout (e.g. `312 ms`) and a present count are visible in monospaced type.
3. Given latency is momentarily unavailable (e.g. reconnecting), when measurement is missing, then the readout shows a neutral placeholder (e.g. `— ms`) rather than a stale value.

---

## L2-014: Codec selection with hardware path
**Traces to:** L1-003

The system must support H.264 and VP9 for WebCodecs, prefer a hardware-accelerated path, and let
the preference be configured (see L2-060); AV1 encode is out of scope for v1.

**Acceptance Criteria:**
1. Given a machine with a hardware H.264 encoder, when the H.264 preference is active, then the hardware encode path is used.
2. Given the VP9 preference and a supporting hardware path, when publishing, then VP9 is used.
3. Given no hardware path for the preferred codec, when publishing, then the system falls back to a supported codec rather than failing to go live.

---

## L2-015: Active-speaker indication
**Traces to:** L1-003

The room must indicate who is currently speaking, on both the main stage and the filmstrip.

**Acceptance Criteria:**
1. Given a participant whose microphone level crosses the speaking threshold, when they speak, then their tile shows a speaking indication.
2. Given a muted participant, when audio is absent, then no speaking indication is shown for them.
3. Given the host is speaking, when shown on the main stage, then a "Speaking" badge is displayed.

---

## L2-016: Microphone control
**Traces to:** L1-004

Each person must be able to mute and unmute their microphone from the in-room dock, with the
control reflecting current state.

**Acceptance Criteria:**
1. Given a live participant, when they activate the mic control, then their microphone toggles between muted and live and the control's pressed state and label update.
2. Given a muted participant, when muted, then no audio frames are transmitted from them and their roster entry shows "muted".
3. Given the mic control, when reached by keyboard, then it is focusable, has an accessible label, and toggles on Enter/Space.

---

## L2-017: Camera control
**Traces to:** L1-004

Each person must be able to turn their camera on and off from the dock; turning it off must stop
transmitting video and show a camera-off indicator to others.

**Acceptance Criteria:**
1. Given a live participant, when they turn the camera off, then video transmission stops and their tile shows a camera-off indicator for all viewers.
2. Given the camera is off, when they turn it back on, then video resumes from the selected device.
3. Given a person joins with camera off (L2-008), when they are in the room, then their tile correctly shows camera-off from the start.

---

## L2-018: Captions toggle in room
**Traces to:** L1-004

Each person must be able to toggle live captions in the room; the control must reflect the
on/off state and the setting must reflect the person's default.

**Acceptance Criteria:**
1. Given captions are available, when a person toggles the captions control, then the caption overlay shows or hides accordingly and the control's pressed state updates.
2. Given the on-device model is unavailable, when the room loads, then the captions control is hidden rather than offering a server-based alternative.
3. Given a person entered with captions on (L2-008), when the room loads, then captions are visible by default.

---

## L2-019: Companion access from the room
**Traces to:** L1-004

The room must give each person access to the companion (Scripture surfacing and the lament
journal) without leaving the gathering, via the dock and the side rail.

**Acceptance Criteria:**
1. Given a live room on a wide viewport, when displayed, then a "Word" dock control and companion links (Scripture, journal) are available.
2. Given a person opens the companion, when they do, then the gathering is not ended and they can return to it.
3. Given the companion links, when the person is a host vs a participant, then the labels match the role context (e.g. "Surface a passage" vs "Follow along in the Word").

---

## L2-020: Presence roster
**Traces to:** L1-004

The room must show a roster of everyone present, marking the host, the current user, muted
participants, and who is speaking.

**Acceptance Criteria:**
1. Given a live room, when the roster is shown, then every present participant is listed with a name/initial and the host is marked "host".
2. Given the current user, when listed, then their own row is distinguished (e.g. "you") and shows their mute state.
3. Given a participant mutes or starts speaking, when their state changes, then their roster indicator updates accordingly.

---

## L2-021: In-room layout and present count
**Traces to:** L1-004

The room must show the present count and adapt its layout responsively: the side rail collapses
on narrower viewports while the stage and dock remain usable.

**Acceptance Criteria:**
1. Given a live room, when displayed, then the header shows the present count alongside the latency readout.
2. Given a viewport width `< 992px`, when the room is shown, then the side rail is hidden and the stage plus dock remain fully usable.
3. Given a viewport width `≥ 992px`, when the room is shown, then the stage and the 320px side rail are shown side by side.
4. Given any supported viewport (XS–XL), when the room is shown, then the control dock stays reachable and does not overflow the viewport horizontally.

---

## L2-022: Send a reaction as a datagram
**Traces to:** L1-005

Any participant must be able to send an "amen"/raised-hand reaction, transmitted as a lightweight
WebTransport datagram.

**Acceptance Criteria:**
1. Given a live participant, when they activate the amen control, then a reaction is sent as a WebTransport datagram (not a reliable stream).
2. Given rapid repeated activation, when reactions are sent, then the client rate-limits them so the room is not flooded.
3. Given the amen control, when reached by keyboard, then it is operable and has an accessible label.

---

## L2-023: Surface reactions to everyone near-instantly
**Traces to:** L1-005

Reactions must appear to all participants near-instantly as ambient "motes", conveying the room
responding as one.

**Acceptance Criteria:**
1. Given a reaction is sent by one participant, when it is received, then a rising mote/animation is shown to all participants promptly (target under ~500 ms on a LAN-class link).
2. Given many reactions arrive at once, when rendered, then they aggregate visually without degrading room frame rate.
3. Given reduced-motion is enabled, when a reaction arrives, then it is conveyed without large motion (e.g. a brief static indication).

---

## L2-024: Aggregate reaction count
**Traces to:** L1-005

The room must show an aggregate reaction count over a recent window so the collective assent is
legible.

**Acceptance Criteria:**
1. Given reactions over the last 60 seconds, when the room is shown, then a rolling count (e.g. "24 raised") is displayed.
2. Given no reactions in the window, when displayed, then the count reads zero without error.
3. Given the window advances, when older reactions age out, then the count decreases accordingly.

---

## L2-025: Generate invite artifacts
**Traces to:** L1-006

The host must be given an invite link, a six-character room code, and a QR code encoding the link.

**Acceptance Criteria:**
1. Given a live room, when the host opens Invite, then a link (`.../r/{CODE}`), the room code, and a QR of the link are shown.
2. Given the room code and the invite link, when both are shown, then the code embedded in the link matches the displayed code.
3. Given the invite surface, when shown, then it states rooms stay small on purpose.

---

## L2-026: Copy invite link and code
**Traces to:** L1-006

The host must be able to copy the invite link (and code) to the clipboard with clear confirmation.

**Acceptance Criteria:**
1. Given the invite dialog, when the host activates "Copy link", then the invite link is written to the clipboard and a confirmation is shown.
2. Given the clipboard API is unavailable or denied, when copy is attempted, then a fallback (selectable text) is offered and no unhandled error occurs.
3. Given the copy control, when reached by keyboard, then it is operable and labelled.

---

## L2-027: Room code format and unpredictability
**Traces to:** L1-006

Room codes must be six characters from an unambiguous alphabet, unique among open rooms, and
generated with sufficient entropy to resist guessing.

**Acceptance Criteria:**
1. Given a new room, when its code is generated, then it is six characters from a defined unambiguous character set and is unique among currently open rooms.
2. Given the space of valid codes, when codes are issued, then they are drawn from a cryptographically strong source such that sequential or adjacent codes are not predictable.
3. Given a room has ended, when its code is later submitted, then it does not resolve to the ended room (see L2-032).

---

## L2-028: Participant leaves the room
**Traces to:** L1-007

A participant must be able to leave via a confirmation step and may rejoin with the same link
while the room remains open.

**Acceptance Criteria:**
1. Given a participant in the room, when they choose Leave, then a confirmation dialog appears explaining they can rejoin with the same link while the room is open.
2. Given the confirmation, when they confirm leaving, then their media stops and they return to the landing screen.
3. Given they left and the room is still open, when they reopen the same link, then they can rejoin.
4. Given the confirmation, when they cancel, then they remain in the room with media uninterrupted.

---

## L2-029: Host ends the gathering
**Traces to:** L1-007

The host must be able to end the gathering for everyone via a clearly destructive confirmation;
ending must close the room so the same link no longer reopens it.

**Acceptance Criteria:**
1. Given the host in the room, when they choose End, then a destructive-styled confirmation states it will close the room for all present and that the same link will not reopen it.
2. Given the confirmation, when the host confirms, then all participants are disconnected and the room is closed.
3. Given the room has ended, when anyone opens the former link/code, then the "room not found" state (L2-032) is shown.
4. Given the confirmation copy, when shown, then it reassures that on-device data (e.g. journal) is not deleted.

---

## L2-030: Reconnect on dropped connection
**Traces to:** L1-007

On a dropped connection, the client must show a reconnecting overlay, pause outbound media, retry
with visible attempts, and offer to leave.

**Acceptance Criteria:**
1. Given a live participant whose connection drops, when the drop is detected, then a reconnecting overlay appears and the camera and microphone are paused so nothing broadcasts while away.
2. Given reconnection is in progress, when retries occur, then the overlay shows attempt progress and the last stable latency.
3. Given reconnection succeeds, when the stream is restored, then the overlay dismisses and prior mute/camera state is restored.
4. Given the person chooses "Leave room" during reconnect, when they confirm, then they exit cleanly to the landing screen.

---

## L2-031: Room full
**Traces to:** L1-007

Because rooms are small by design, a join attempt on a room at capacity must be refused with a
clear, non-error explanation and next steps.

**Acceptance Criteria:**
1. Given a room at its capacity limit, when a person attempts to join, then a "room is full" state explains rooms are small on purpose and offers "Try again" and "Host a room".
2. Given the room later has space, when the person retries, then they can join.
3. Given the capacity limit, when enforced, then it is applied server-side and cannot be bypassed by the client.

---

## L2-032: Room not found
**Traces to:** L1-007

A code or link that matches no open room must produce a clear "not found" state naming the likely
causes and offering recovery.

**Acceptance Criteria:**
1. Given a well-formed code that matches no open room, when submitted, then a "that room isn't open" state names the code and explains it may have ended or been mistyped.
2. Given the not-found state, when shown, then it offers "Try another code" and "Host your own room".
3. Given an unknown non-room URL within the app, when opened, then a 404 state returns the person to the gathering/home.

## Subsystem B — Word

## L2-033: Name a theme
**Traces to:** L1-008

Scripture surfacing must accept a plain-language theme by typing, with quick-pick suggestion
chips, framed as everyday words.

**Acceptance Criteria:**
1. Given the Scripture screen, when it loads, then a theme input and a set of quick-pick chips (e.g. fear, gratitude, waiting) are shown.
2. Given the person selects a chip, when it is chosen, then the theme input is populated with that theme.
3. Given a theme is entered, when the person submits, then a passage is surfaced (L2-034) or the no-match state (L2-036) is shown.
4. Given theme text, when it is echoed back in the result, then it is rendered as inert text (L2-073).

---

## L2-034: Surface a passage from a local verse index
**Traces to:** L1-008

The system must match the theme against a verse index stored on the device and return an existing
passage — retrieved, never generated — with the reference shown.

**Acceptance Criteria:**
1. Given a theme with a match, when surfaced, then a passage and its reference are shown in the serif register, drawn from the local verse index.
2. Given a surfaced passage, when shown, then supporting copy states the passage was matched from a local index and retrieved (chosen, not composed).
3. Given the surfacing runs, when it executes, then no network request is made and matching happens entirely on-device (L2-070).

---

## L2-035: Passage result actions
**Traces to:** L1-008

A surfaced passage must offer to surface another, read in context, and save, plus a plain "why
this surfaced" explanation.

**Acceptance Criteria:**
1. Given a surfaced passage, when shown, then "Surface another", "Read in context", and "Save" actions are available.
2. Given "Surface another", when activated, then a different fitting passage for the same theme is shown where the index allows.
3. Given "Save", when activated, then the passage is saved to on-device storage and remains available offline.
4. Given a recent-themes list, when a theme has been surfaced before, then it appears in "Recent" stored only on the device.

---

## L2-036: No passage matched
**Traces to:** L1-008

When the local index has no match, the system must say so plainly and suggest plainer alternative
words rather than fabricating a reference.

**Acceptance Criteria:**
1. Given a theme with no index match, when surfaced, then a "no passage matched that yet" state is shown and no passage is fabricated.
2. Given the no-match state, when shown, then it suggests plainer alternative themes as chips and offers to write in the journal instead.
3. Given the no-match state, when shown, then it reaffirms the index is local and made zero requests.

---

## L2-037: Recent themes are device-local
**Traces to:** L1-008

Recently surfaced themes must be stored on the device only and never transmitted.

**Acceptance Criteria:**
1. Given surfaced themes, when the person returns to the Scripture screen, then recent themes are listed from device storage.
2. Given recent themes exist, when the surfacing runs, then no theme history is sent off-device.
3. Given the person clears local data, when cleared, then recent themes are removed from the device.

---

## L2-038: Journal editor with on-device indicator
**Traces to:** L1-009

The journal must provide a free-writing editor showing a live word count and a persistent
on-device/private indicator, with copy affirming no one else reads it.

**Acceptance Criteria:**
1. Given the journal, when the person types, then a live word count updates and an "on-device" indicator is visible.
2. Given the journal, when shown, then a privacy seal states the entry is private, on-device, and never transmitted.
3. Given text entry, when the person writes, then no keystroke or content is transmitted off-device (L2-070).

---

## L2-039: Journal entries persist on-device only
**Traces to:** L1-009

Journal entries must be stored on the device only and listed as prior ("Earlier") entries, clearly
marked as local.

**Acceptance Criteria:**
1. Given prior entries, when the journal loads, then an "Earlier" list shows them with dates, marked "only on this device".
2. Given a saved entry, when the person reloads the app on the same device, then the entry is still present.
3. Given entries exist, when any journal action runs, then no entry content leaves the device.

---

## L2-040: Ask for a reflection (on-device)
**Traces to:** L1-009

The journal must let the person request a reflection generated on the device from their own words,
returned with an explicit on-device/zero-requests affirmation.

**Acceptance Criteria:**
1. Given a non-empty entry, when the person requests a reflection, then a reflection is generated on-device and shown, with copy stating it was written on the device and zero requests left the machine.
2. Given the reflection is generating, when it runs, then a visible in-progress state is shown and no network request is made.
3. Given the on-device model is unavailable, when the person opens the journal, then the reflection action is hidden (L2-066) — writing and saving still work.

---

## L2-041: Reflections are gentle and non-authoritative
**Traces to:** L1-009

Reflections must be gentle, grounded, and never authoritative or pastoral-directive, and must
leave the last word with the person; any Scripture referenced must be presented as illustrative.

**Acceptance Criteria:**
1. Given a reflection, when shown, then accompanying copy communicates it is gentle and never final — the last word is the person's.
2. Given a reflection references Scripture, when shown, then the reference is offered as an invitation to sit with (not a command) and framed as illustrative.
3. Given the reflection content, when produced, then it must not issue directive spiritual instruction or definitive judgment about the person's situation.

---

## L2-042: Reflection feedback
**Traces to:** L1-009

The person must be able to keep a reflection with the entry or dismiss it as not helpful, all on
the device.

**Acceptance Criteria:**
1. Given a reflection, when the person chooses "Keep with entry", then it is stored alongside the entry on-device.
2. Given a reflection, when the person chooses "Not helpful", then it is dismissed without being stored and no feedback is transmitted.
3. Given either choice, when made, then it triggers no network request.

---

## L2-043: Clear the current entry
**Traces to:** L1-009

The journal must let the person clear the current draft.

**Acceptance Criteria:**
1. Given text in the editor, when the person activates Clear, then the editor content is emptied.
2. Given the editor is empty, when Clear is shown, then it does not throw or navigate away.

---

## L2-044: Empty journal state
**Traces to:** L1-009

With no entries, the journal must present an inviting empty state affirming privacy and offering
to start writing.

**Acceptance Criteria:**
1. Given no journal entries, when the journal opens, then an empty state invites the first entry and affirms it is private and on-device.
2. Given the empty state, when the person activates "Start writing", then the editor opens ready for input.

---

## L2-045: On-device transcription
**Traces to:** L1-010

Live captions must transcribe spoken audio on the device in real time, with no audio or transcript
transmitted off-device.

**Acceptance Criteria:**
1. Given captions enabled and speech occurring, when a person speaks, then transcribed text appears in near real time.
2. Given transcription runs, when it executes, then neither audio nor transcript is transmitted off-device (L2-070).
3. Given the on-device model is unavailable, when captions are requested, then captions are hidden rather than routed to a cloud transcriber.

---

## L2-046: Caption display
**Traces to:** L1-010

The room must display captions attributed to the speaker, distinguishing finalized text from the
in-progress tail, and must be legible over the video.

**Acceptance Criteria:**
1. Given captions enabled, when a caption is shown, then it is attributed to the current speaker and the in-progress portion is visually distinguished from finalized text.
2. Given captions over video, when displayed, then contrast keeps them legible against the stage (L2-084).
3. Given captions, when displayed, then they are announced to assistive technology via a live region (L2-083).

---

## L2-047: Caption language selection
**Traces to:** L1-010

The person must be able to choose the caption language from the supported set in settings.

**Acceptance Criteria:**
1. Given caption settings, when the language select is shown, then the supported languages are listed.
2. Given a chosen language, when captions run, then transcription targets that language.
3. Given a selected language, when saved, then the choice persists on the device (L2-063).

---

## L2-048: Captions default persistence
**Traces to:** L1-010

A person's captions on/off preference must persist across sessions on the device and seed both the
green room and the in-room toggle.

**Acceptance Criteria:**
